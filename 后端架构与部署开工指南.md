# 后端架构与部署开工指南

> 目的：明天打开这份文档，可以快速知道后端当前状态、下一步方向、优先级和可直接开工的任务。

## 一句话结论

当前后端架构可以支撑 MVP 演示：`FastAPI app/` 负责 HTTP 接口，`agent/` 负责智能体运行时，`Project` service 作为中间门面编排 session、scheduler、workspace、metrics。

但它还不是完全低耦合、可多人并发、可独立替换 agent 的成熟架构。明天优先把后端和 agent 的边界收紧，再做服务器部署验证。

## 当前架构现状

后端主链路：

```text
前端
  -> FastAPI routes
  -> app/services/project.py: Project
  -> agent/scheduler.py: Scheduler
  -> agent/actions.py / agent/llm.py / agent/tools/*
  -> SSE 返回执行过程
```

当前分层：

| 层级 | 位置 | 职责 |
|---|---|---|
| API 层 | `backend/app/api/routes/` | HTTP 路由、SSE 输出、请求响应转换 |
| Schema 层 | `backend/app/schemas/` | Pydantic 请求模型 |
| Service 层 | `backend/app/services/project.py` | workspace、session、scheduler、metrics、shadow repo 编排 |
| Agent 引擎 | `backend/agent/` | ReAct 循环、LLM 调用、工具执行、session 持久化 |
| 配置层 | `backend/app/core/` | Settings、CORS、临时目录常量 |

## 需要记住的关键判断

1. 获取和启动 agent 的入口已经在 service 层：`Project.start_chat()`。
2. agent 核心逻辑不在 service 层，而是在 `backend/agent/`。
3. 当前架构方向是对的：API 不应该直接写 agent 逻辑，应该通过 `Project` 作为 facade。
4. 目前耦合还偏高：部分路由直接访问 agent 内部私有字段，例如 `session._transcripts`、`session._sandbox`。
5. 当前只有一个全局 `Scheduler`，一次只能运行一个 session；比赛 demo 可以接受，公网多人使用会有并发问题。

## 明天优先任务

### P0：先保证后端边界清晰

目标：让 API 层只调用 service，不直接碰 agent 内部实现。

建议改造：

1. 在 `Project` 中新增方法：
   - `checkout_session(...)`
   - `interrupt_session(session_id)`
   - `respond_to_pending(transcript_id, response)`
   - `get_session_status(session_id)`

2. 把 `backend/app/api/routes/sessions.py` 中的 checkout 逻辑下沉到 `Project`。

当前风险点：

```python
session._transcripts
session._sandbox
from agent.tools.task import reconstruct_tasks
```

这些不应该出现在 route 层。

### P1：修正 session status 语义

当前 `/api/session/{sid}/status` 返回的是全局 scheduler 状态：

```python
return {"running": project.scheduler.state != "idle"}
```

这会导致 A session 正在运行时，查询 B session 也显示 running。

建议改成：

```python
return {"running": project.scheduler.is_running_session(sid)}
```

或者封装到：

```python
project.get_session_status(sid)
```

### P1：降低 agent 对 app 的反向依赖

现在 `backend/agent/` 内部仍然 import 了：

```python
from app.core.config import TMP_DIR
```

涉及文件包括：

- `backend/agent/session.py`
- `backend/agent/scheduler.py`
- `backend/agent/shadow_repo.py`
- `backend/agent/tools/grep.py`
- `backend/agent/tools/glob.py`
- `backend/agent/tools/task.py`

建议做法：

1. 新增 `backend/agent/config.py`，定义 agent 自己需要的默认配置。
2. 或者把 `tmp_dir` 从 `Project` 初始化时注入到 agent 对象。
3. 短期为了稳，可以先新增 `agent/config.py` 并让这些文件依赖它。

### P1：明确部署模型

如果只是比赛演示：

- 单进程 FastAPI
- 单 workspace
- 单 Scheduler
- 一个用户现场操作

这样基本够用。

如果要公网 demo：

- 需要限制并发，避免多个用户同时跑 agent。
- 需要保护 `.env`、API Key、workspace 写权限。
- 需要确认 `.tmp/`、SQLite metrics、shadow repo 在服务器上有稳定可写目录。
- 建议加一个简单的访问口令或 Basic Auth，避免外部用户滥用模型额度。

## 部署前检查清单

### 环境变量

检查 `backend/.env`：

```text
LLM_API_KEY=
LLM_BASE_URL=
LLM_MODEL_ID=
LLM_TIMEOUT=
AGENT_WORKSPACE=
BROWSER_HEADLESS=1
LLM_METRICS_DB_PATH=.tmp/ai_metrics.sqlite3
```

注意：不要提交真实 `.env`。

### 依赖

后端：

```bash
cd backend
uv sync
uv run playwright install chromium
```

前端：

```bash
cd frontend
npm install
npm run build
```

### 服务器关注点

- Python 版本是否满足项目要求。
- Playwright Chromium 是否能安装。
- 如果服务器没有图形界面，`BROWSER_HEADLESS=1`。
- `.tmp/` 是否可写。
- API Key 是否通过服务器环境变量注入。
- 前端是否正确代理或配置后端 API 地址。
- 是否需要 Nginx 反向代理。
- 是否需要 HTTPS。
- 是否要限制访问，避免公开消耗 API 额度。

## 和 agent 同学协作的边界建议

后端负责：

- HTTP API
- SSE 协议
- session 生命周期
- workspace 切换
- metrics 查询
- 部署和服务可用性
- 把 agent 能力包装成稳定 service 接口

agent 同学负责：

- `backend/agent/` 内部运行时
- ReAct 循环
- LLM 调用策略
- tools 和 skills
- prompt
- subagent
- sandbox 行为

双方约定接口：

```python
Project.start_chat(session_id, question, max_steps)
Project.get_history(session_id)
Project.get_recovery_state(session_id)
Project.respond_to_pending(transcript_id, response)
Project.interrupt_session(session_id)
Project.checkout_session(session_id, ...)
```

后端不要直接依赖 agent 内部字段；agent 不要反向 import `app`。

## 明天可以直接做的小改动

### 1. 修 status

文件：`backend/app/api/routes/sessions.py`

把：

```python
return {"running": project.scheduler.state != "idle"}
```

改成：

```python
return {"running": project.scheduler.is_running_session(sid)}
```

### 2. 封装 interrupt

在 `Project` 里加：

```python
async def interrupt_session(self, session_id: str) -> bool:
    self.get_session(session_id)
    return await self.scheduler.interrupt()
```

route 改成只调用：

```python
ok = await project.interrupt_session(sid)
```

### 3. 封装 respond

在 `Project` 里加：

```python
def respond_to_pending(self, transcript_id: str, response: dict) -> None:
    self.scheduler.resolve(transcript_id, response)
```

route 改成只调用 service。

### 4. 把 checkout 逻辑从 route 移到 Project

这是明天最值得做的结构优化。做完后，`sessions.py` 会明显变薄，后端和 agent 的边界也更清楚。

## 当前风险排序

| 风险 | 影响 | 优先级 |
|---|---|---|
| 单 Scheduler 只能跑一个 session | 多人访问、并发 demo 会冲突 | 高 |
| route 层访问 agent 私有字段 | agent 同学改内部实现会破坏后端 | 高 |
| agent 反向依赖 app config | 不利于 agent 独立拆包 | 中 |
| `RespondRequest.response` 是裸 dict | 后续人机交互协议容易混乱 | 中 |
| `.tmp/`、SQLite、shadow repo 都依赖本地文件系统 | 部署时需要稳定持久化目录 | 中 |
