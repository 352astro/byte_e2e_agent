演示流程：
PM 自然语言输入需求 -> agent 处理（澄清 - 方案 - 模块定位 - 代码生成 - Lint/单侧 - 上传远程仓库） -> 提交 PR
1. 要求能现场完成 P1 级别需求

6. 澄清深度：能识别 PM 模糊需求中的歧义并主动追问，而不是自行假设硬编生成。

## 评分标准
1. 技术深度与创新性 30% ：Skill / Agent / Orchestrator 分层是否原创且自洽；上下文召回策略是否超越 RAG 暴力切片；澄清 Agent 对模糊需求的拆解深度
2. 技术实现与工程完整度 25% ：系统能在 Conduit 实仓上稳定生成可提测代码；Lint / 单测 / 集成测试绿；具备 token、时延、成本监控；至少 1 条端到端链路现场可演示
3. 业务价值与场景契合度 20% ：PM 视角的对话流是否自然、错误反馈是否友好、操作路径是否符合直觉；是否能识别歧义并主动追问；新增需求模式接入成本（改文件数、生效时间）；是否沉淀业务上下文反哺后续需求
4. 代码质量与文档 10% ：系统自身代码通过 Lint；README / 架构图 / 关键模块注释齐全；提交记录可追溯
5. 数据与合规性 5% ：none
6. 项目材料完整度 10% ：按 §八清单全量交付：在线 Demo / 3–8min 演示视频 / 公开仓库 + README / 架构图 / AI 使用说明 / 工程难点说明 / 部署说明

## 提交材料清单
1. 基础信息
    - 项目名称（统一保持本课题命名）
    - 团队名称、成员名单（列出成员姓名、学校、专业、角色）
    - 分工说明
2. 功能说明 
    - 端到端使用流程（用 5–8 句写清 PM 从输入需求到拿到可提测 PR 的完整流程）
3. 交付材料
    - 在线 Demo 链接（可直接访问；如需登录请提供体验账号或录屏替代）

4. 技术说明
    - 系统架构图（展示前端 / 后端 / 模型层 / Skill 注册表 / Orchestrator / sandbox-repo 调用关系）

    - 核心技术栈（分别说明前端 / 后端 / 数据库 / 中间件 / 部署环境 / 云资源等）、
    - 项目亮点 / 创新点（上述的加分的）
